Prompt Engineering 进阶

ReAct 模式与自我反思

▼
💼 对应招聘要求
JD 关键词：了解 ReAct/Reflection 等推理模式、能优化 Agent 行为
面试考点：ReAct 原理实现、Self-Critique 应用场景
📚 学习内容 （点击展开详情）
  • 深入理解 ReAct (Reasoning + Acting) 模式
    权威来源： ReAct: Synergizing Reasoning and Acting in Language Models (Yao et al., 2022)

    核心概念： ReAct 将推理（Reasoning）和行动（Acting）交织在一起，形成循环：
    Thought → Action → Observation → Thought → ...

    为什么有效：
    - Thought：让模型"说出"推理过程，减少幻觉
    - Action：执行具体操作（搜索、计算、调用 API）
    - Observation：获取真实世界反馈，而非靠猜测

    代码示例：
    Question: 奥巴马的妻子是谁？她出生在哪个城市？
    Thought 1: 我需要先查询奥巴马的妻子是谁
    Action 1: Search[奥巴马 妻子]
    Observation 1: 米歇尔·奥巴马是巴拉克·奥巴马的妻子...
    Thought 2: 米歇尔·奥巴马是答案的第一部分，现在查她的出生地
    Action 2: Search[米歇尔·奥巴马 出生地]
    Observation 2: 米歇尔·奥巴马出生于伊利诺伊州芝加哥
    Thought 3: 我现在知道完整答案了
    Action 3: Finish[米歇尔·奥巴马，芝加哥]

    实践建议： 在 Prompt 中明确要求模型按 Thought/Action/Observation 格式输出
    预计时间： 1.5 小时阅读论文 + 1 小时动手实现
  • 学习 Self-Critique 和反思机制
    权威来源： Reflexion: Language Agents with Verbal Reinforcement Learning (Shinn et al., 2023)

    核心概念： Reflexion 让 Agent 从失败中学习，通过语言反馈进行自我改进：
    1. Actor：执行任务
    2. Evaluator：评估结果（成功/失败/部分成功）
    3. Self-Reflection：分析失败原因，生成改进建议
    4. Memory：存储反思结果，下次避免同样错误

    Self-Critique Prompt 模板：
    我刚刚给出的回答是：[回答内容]

    请检查这个回答：
    1. 有没有事实错误？
    2. 逻辑是否连贯？
    3. 是否回答了用户的真正问题？
    4. 有没有遗漏重要信息？

    如果发现问题，请给出修正后的回答。

    实践建议： 在关键任务（代码生成、数据分析）中加入自我检查步骤
    预计时间： 1 小时阅读 + 1.5 小时实现带反思的 Agent
  • 实践：优化 Agent 的 System Prompt
    权威来源： Anthropic CoT 最佳实践

    ReAct Prompt 模板：
    你是一个问题解决专家。对于每个问题，请按以下格式思考和行动：

    Thought: [你的推理过程]
    Action: [要执行的动作，格式为 ToolName[参数]]
    Observation: [工具返回的结果，由系统填充]
    ... (重复直到得出答案)
    Thought: 我现在知道答案了
    Action: Finish[最终答案]

    可用工具：
    - Search[query]: 搜索网络信息
    - Calculate[expression]: 执行数学计算
    - Lookup[term]: 在当前文档中查找术语

    优化技巧：
    1. 明确工具列表和使用格式
    2. 给出 1-2 个 Few-shot 示例
    3. 设定最大循环次数防止死循环
    4. 要求在 Finish 前确认答案完整性

    预计时间： 2 小时迭代优化 Prompt

    - Toolformer 论文 (Schick et al., 2023)

    核心概念： Function Calling 让 LLM 不仅生成文本，还能结构化地调用外部函数

    OpenAI 格式：
    {
    "name": "get_weather",
    "description": "获取指定城市的天气",
    "parameters": {
    "type": "object",
    "properties": {
    "city": {"type": "string", "description": "城市名称"}
    },
    "required": ["city"]
    }
    }

    Claude 差异： Claude 使用 tool_use 格式，支持更复杂的嵌套参数
    国产模型差异： GLM-4、Qwen 基本兼容 OpenAI 格式，但细节处理可能不同

    预计时间： 1.5 小时阅读文档 + 对比测试
  • 学习 JSON Schema 定义工具接口
    权威来源： JSON Schema 官方教程

    核心概念： JSON Schema 是工具参数定义的标准格式

    常用类型：
    "type": "string" // 字符串
    "type": "number" // 数字
    "type": "boolean" // 布尔值
    "type": "array" // 数组，需要 items 定义元素类型
    "type": "object" // 对象，需要 properties 定义属性
    "enum": ["A", "B"] // 枚举，限定可选值

    复杂示例：
    {
    "name": "search_products",
    "parameters": {
    "type": "object",
    "properties": {
    "keywords": {"type": "string"},
    "category": {"type": "string", "enum": ["electronics", "clothing", "books"]},
    "price_range": {
    "type": "object",
    "properties": {
    "min": {"type": "number"},
    "max": {"type": "number"}
    }
    },
    "sort_by": {"type": "string", "enum": ["price", "rating", "newest"]}
    },
    "required": ["keywords"]
    }
    }

    预计时间： 1 小时学习 + 定义 5 个工具 Schema
  • 研究 LangChain Tools 的实现
    权威来源： LangChain Tools 文档

    核心概念： LangChain 提供三种定义工具的方式

    方式一：@tool 装饰器（最简单）
    from langchain.tools import tool

    @tool
    def get_weather(city: str) -> str:
    """获取指定城市的当前天气。"""
    return f"{city}的天气是晴天，25°C"

    方式二：StructuredTool（复杂参数）
    from langchain.tools import StructuredTool
    from pydantic import BaseModel

    class SearchInput(BaseModel):
    query: str
    max_results: int = 5

    search_tool = StructuredTool.from_function(
    func=search_web,
    name="web_search",
    description="搜索网络信息",
    args_schema=SearchInput
    )

    方式三：继承 BaseTool（完全控制）
    from langchain.tools import BaseTool

    class CalculatorTool(BaseTool):
    name = "calculator"
    description = "执行数学计算"

    def _run(self, expression: str) -> str:
    return str(eval(expression))

    async def _arun(self, expression: str) -> str:
    return self._run(expression)

    预计时间： 2 小时实现 3 个自定义 Tool

  • 实现：接收问题 → 搜索网页 → 总结内容 → 回答问题
    核心代码示例：
    from langchain.agents import create_react_agent, AgentExecutor
    from langchain_openai import ChatOpenAI
    from langchain_community.tools import DuckDuckGoSearchRun
    from langchain import hub

    # 初始化
    llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
    tools = [DuckDuckGoSearchRun()]
    prompt = hub.pull("hwchase17/react")

    # 创建 Agent
    agent = create_react_agent(llm, tools, prompt)
    executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

    # 执行查询
    result = executor.invoke({"input": "2024年AI领域最重要的突破是什么？"})
    print(result["output"])

    优化技巧：
    1. 设置 max_iterations 防止死循环
    2. 添加搜索结果的后处理（去重、截断）
    3. 使用流式输出提升体验

    Agent A → Agent B → Agent C → 输出

    2. 并行（Parallel）
    ┌→ Agent A ─┐
    输入 ──┼→ Agent B ──┼→ 合并 → 输出
    └→ Agent C ─┘
    适用：独立任务并发，如多角度分析

    Supervisor Agent
    ├── Worker A
    ├── Worker B
    └── Worker C
    适用：任务分配和协调

    4. 循环（Loop/Iterative）
    Agent A ↔ Agent B (多轮对话直到收敛)
    适用：辩论、代码审查、迭代优化

    draft: str
    final: str

    # 研究逻辑
    return {"research": "研究结果..."}

    # 写作逻辑
    return {"draft": f"基于 {state['research']} 写的草稿"}

    def reviewer(state):
    # 审核逻辑
    return {"final": "修改后的最终版本"}

    workflow.add_node("writer", writer)
    workflow.add_node("reviewer", reviewer)

    workflow.add_edge("writer", "reviewer")

    from abc import ABC, abstractmethod

    class LLMProvider(ABC):
    @abstractmethod
    def chat(self, messages: list) -> str:
    pass

    @abstractmethod
    def stream(self, messages: list):
    pass

    class OpenAIProvider(LLMProvider):
    def __init__(self, api_key, base_url=None, model="gpt-4o-mini"):
    self.client = OpenAI(api_key=api_key, base_url=base_url)
    self.model = model

    def chat(self, messages):
    response = self.client.chat.completions.create(
    model=self.model,
    messages=messages
    )
    return response.choices[0].message.content

    Q1: RAG 和 Fine-tuning 的区别？什么时候用哪个？
    RAG：
    - 适合知识更新频繁的场景
    - 无需重新训练，成本低
    - 可追溯答案来源

    Fine-tuning：
    - 适合改变模型行为/风格
    - 处理特定领域术语
    - 提升特定任务性能

    选择：大多数知识问答场景优先 RAG

    Q2: Agent 和传统 Pipeline 的区别？
    Pipeline：
    - 固定流程，A → B → C
    - 可预测，易调试

    Agent：
    - 动态决策，自主规划
    - 灵活但不可预测
    - 需要更多 guard rails

    Q3: 如何优化 RAG 检索效果？
    1. Chunk 策略：语义分割 vs 固定大小
    2. Hybrid Search：向量 + BM25
    3. Reranker：Cohere/BGE 重排
    4. Query Expansion：HyDE、多查询
    5. 元数据过滤：预筛选相关文档
